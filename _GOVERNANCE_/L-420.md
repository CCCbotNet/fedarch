## 📋 Learning Statement

> **"You MUST document otherwise 'drift' will occur and your GREAT IDEA will be forgotten!!"**
> — @GTM (yonks｜🤖🏛️🪙｜Jason Younker ♾️)

**@GTM:MANTRA — 1:DOCUMENT → 2:ITERATE → 3:AUTOMATE.**

Governance standards MUST be defined first (DOCUMENT), manually
executed while tooling is built (ITERATE), then automated when
infrastructure supports it (AUTOMATE). Do NOT over-engineer step 3
before step 1 is complete.

---

## 📋 @GTM:MANTRA — 3-Step Framework

| Step | Name | Description | Status |
|:----:|------|-------------|:------:|
| **1** | **DOCUMENT** | Define the standard in writing — rules, learnings, BPs, definitions | 🔒 ALWAYS FIRST |
| **2** | **ITERATE** | Execute manually while tooling is built — validate the standard works | 🔄 WHILE BUILDING |
| **3** | **AUTOMATE** | Encode into tooling when infrastructure supports it | ⬜ WHEN READY |

### The Guardrail

> **Do NOT over-engineer step 3 before step 1 is complete.**

| Anti-Pattern | Why It Fails |
|--------------|-------------|
| Build automation before standard is defined | Automates the wrong thing |
| Skip iteration — go straight to automation | No validation that standard works |
| Document without iterating | Standard never tested in practice |
| Automate without documenting | Black box — no one knows what it does |

## 📋 The Drift Problem

### What Is Drift?

| Drift Type | Description | Example |
|------------|-------------|---------|
| **Memory drift** | Agent forgets the rule across sessions | Surge ×4 L-209 failures |
| **Interpretation drift** | Different agents interpret the rule differently | "PRE GH PUSH or DEEP FULL?" |
| **Priority drift** | Rule loses urgency without documentation | "I'll check L-209 next time" |
| **Entropy drift** | Without energy (documentation), systems move toward disorder | Undocumented governance = inconsistency |

### Why "GREAT IDEA" Matters

> **The quote says "GREAT IDEA" — not just "idea."**

| Implication | Detail |
|-------------|--------|
| Great ideas DO happen in #FedArch | L-209, L-217, L-420 — all discovered in real sessions |
| Without documentation, great ideas die | Not from lack of quality — from lack of persistence |
| Documentation = the bridge | Between "discovered in session" and "permanent governance" |
| The cost is HIGH | A great idea lost = governance gap = #BadAgent = drift |

## 📋 Corollary — Persistent Context Injection

> **Documentation = persistent context injection.**
> A documented rule is injected into every future session.
> An undocumented rule exists only in the session where it was discovered.
> **Documentation is a #ContextVolley to your future self.**

*(Discovered by MiMo 🧪 — W13 D1 feedback — ⭐⭐⭐⭐⭐)*

### What This Means

| Documentation State | Effect on Future Sessions |
|--------------------|--------------------------|
| Rule in SharedKernel | Every agent, every instance = rule present ✅ |
| Rule in workspace prompt | Every session in that workspace = rule present ✅ |
| Rule discussed but NOT documented | Only the session where discussed = rule present ⚠️ |
| Rule NOT documented | Next session = rule gone. Drift begins. ❌ |

### Reframing the 3 Steps

| Step | Without Corollary | With Corollary |
|------|-------------------|----------------|
| DOCUMENT | "Write it down" | "Inject context for all future sessions" |
| ITERATE | "Execute manually" | "Verify the injected context works" |
| AUTOMATE | "Build tooling" | "Tooling = automated context injection" |
